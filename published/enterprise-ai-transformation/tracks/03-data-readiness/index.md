# Data Readiness

Data readiness is the most common reason AI programs stall after initial pilots. A model that performs well in a controlled experiment often degrades sharply in production — not because the model is wrong, but because the data feeding it is inconsistent, inaccessible, undocumented, or structured for the system that produced it rather than the system that needs to consume it. Most organizations have more data than they think they need and less usable data than they realize. Closing that gap is not primarily a technology problem; it is an organizational one, requiring governance structures, ownership accountabilities, and engineering investment that many enterprises have deferred.

The challenge has intensified with the shift to large language models and agentic AI. Earlier AI work — forecasting models, recommendation engines, fraud detectors — could be built on clean, narrow datasets assembled for a specific purpose. Generative and agentic systems work across much broader data surfaces: unstructured documents, real-time feeds, proprietary knowledge bases, third-party integrations. Each new surface introduces new quality, lineage, and compliance questions. An AI agent that can read customer emails and update CRM records is not a narrow analytics problem; it requires the organization to have answered fundamental questions about data ownership, access control, and audit logging that many have not yet addressed systematically.

This track covers the six components that determine whether an organization's data can actually power AI at scale. The [core framework](/enterprise-ai-transformation/tracks/03-data-readiness/02-framework/index.md) works through each in sequence: [Data Quality](/enterprise-ai-transformation/tracks/03-data-readiness/02-framework/01-data-quality.md) (dimensions, failure modes, measurement), [Data Governance](/enterprise-ai-transformation/tracks/03-data-readiness/02-framework/02-data-governance.md) (ownership, policy, bias controls), [Access & Integration](/enterprise-ai-transformation/tracks/03-data-readiness/02-framework/03-access-and-integration.md) (silo patterns, integration architectures), [Lineage & Metadata](/enterprise-ai-transformation/tracks/03-data-readiness/02-framework/04-lineage-and-metadata.md) (provenance, active metadata), [Infrastructure Readiness](/enterprise-ai-transformation/tracks/03-data-readiness/02-framework/05-infrastructure-readiness.md) (MLOps, LLMOps, AgentOps maturity), and [Security & Compliance](/enterprise-ai-transformation/tracks/03-data-readiness/02-framework/06-security-and-compliance.md) (threat categories, privacy landscape). Seven practitioner guides cover the operational execution: synthetic data generation, data contracts, data mesh governance, data audits and automated quality governance, historical data debt remediation, master data management, and data labeling programs.

The assessment section provides two tools — an [AI Readiness Assessment Framework](/enterprise-ai-transformation/tracks/03-data-readiness/04-assessment-and-measurement/ai-readiness-assessment-framework.md) for scoring current-state maturity across the six components, and a [Total Cost of Data Debt](/enterprise-ai-transformation/tracks/03-data-readiness/04-assessment-and-measurement/total-cost-of-data-debt.md) model for quantifying the business case for remediation investment.

Leaders looking for the decision-level picture before going deeper should start with the [executive summary](/enterprise-ai-transformation/tracks/03-data-readiness/01-executive-summary.md), which distills the core argument and the sequence of investments most organizations need to make to move from data-constrained to data-ready for AI.
